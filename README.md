## Unified Background Tasks Runbook

This project now uses a unified background tasks model powered by `APScheduler`.
Long-running jobs are server-bound and persisted, so they can continue across page switches and refreshes.

### Unified Task APIs

- `POST /api/tasks`: create a task (`task_type`, `payload`, optional `idempotency_key`)
- `GET /api/tasks/<task_id>`: fetch one task status
- `GET /api/tasks?task_type=&status=&limit=`: list tasks
- `POST /api/tasks/<task_id>/cancel`: request cancel

### Task Status Semantics

- `queued`: accepted and waiting to run
- `running`: currently executing
- `completed`: finished successfully
- `failed`: finished with error
- `cancelled`: cancelled by user/request
- `interrupted`: was running when process stopped; recovered on restart path

### Restart Recovery Behavior

On service startup:

- `queued` tasks are re-scheduled
- `running` tasks are marked `interrupted` and handled by recovery policy
- terminal tasks (`completed`, `failed`, `cancelled`) remain as history records

### Single-Host Runtime Recommendations

- Keep one long-lived process via `python web/app.py`
- Avoid frequent restarts during heavy jobs
- Poll task progress through `/api/tasks` rather than page-local state
- Keep worker/thread concurrency moderate to avoid starving request handling
